# ✅ Ответы и разбор

> Не подглядывай, пока не прошёл весь тест! Считай баллы честно.

## Раздел A

**A1.** Протокол — набор правил обмена данными. Три составляющие: **синтаксис** (формат/структура сообщения), **семантика** (смысл полей и какие действия предпринимать), **тайминг/порядок** (кто говорит первым, чего ждать). *(Урок 1)*

**A2.** **Хост** — конечная точка, источник или получатель данных (ноутбук, сервер). **Узел** — любое устройство в сети, включая промежуточные. Пример узла-не-хоста: **маршрутизатор** или коммутатор. *(Урок 1)*

**A3.** **Hop** — один «прыжок» пакета от одного узла (маршрутизатора) к следующему. **Store-and-forward** — маршрутизатор полностью принимает пакет в буфер, читает заголовок, решает куда слать, и отправляет дальше. *(Урок 2)*

**A4.** **PDU** (Protocol Data Unit) — название порции данных на конкретном уровне. Transport → **сегмент** (TCP) / датаграмма (UDP); Internet → **пакет**; Link → **кадр**. *(Уроки 4, 6)*

**A5.** **Инкапсуляция** — у отправителя, данные идут **сверху вниз**, каждый уровень добавляет свой заголовок. **Деинкапсуляция** — у получателя, **снизу вверх**, каждый уровень снимает свой заголовок. *(Урок 6)*

**A6.** **Overhead** — служебные (не-полезные) байты, которые передаются ради доставки: заголовки уровней. Это «налог» на каждый пакет. *(Уроки 2, 6)*

**A7.** **RTT** (Round-Trip Time) — время прохождения сигнала «туда и обратно» (то, что показывает `ping`). *(Урок 2)*

**A8.** **Best-effort** — модель доставки «постараюсь, но не гарантирую»: пакеты могут потеряться, задержаться, прийти не по порядку или продублироваться; сеть не даёт гарантий. *(Урок 2)*

## Раздел C

**C1. Неверно.** Каждый маршрутизатор знает только **следующий шаг** для данного адреса назначения, а не весь путь. *(Урок 2)*

**C2. Неверно.** Это **логическая иллюзия**. Физически данные идут вниз по стеку A → по среде → вверх по стеку B. «Разговор равных уровней» обеспечивают нижние уровни. *(Урок 3)*

**C3. Неверно.** Задержку **распространения** ограничивает скорость света — деньгами/каналом её не победить. Помогает только физическая близость (CDN). Более быстрый канал уменьшает задержку **передачи**, но не распространения. *(Урок 2)*

**C4. Верно.** Пакеты независимы, выделенной линии нет, сеть не хранит состояние разговора — поэтому каждый пакет несёт адрес сам (независимая маршрутизация). *(Урок 2)*

**C5. Неверно.** Trailer (FCS/контрольная сумма кадра) добавляет **канальный уровень (L2, Ethernet)**, а не IP. *(Урок 6)*

**C6. Верно.** TCP/IP выросла из практики (ARPANET): рабочий код раньше формального описания — одна из причин победы над OSI. *(Урок 5)*

## Раздел E

**E1.** *(Урок 6)*
- **Application:** данные приложения (HTTP) — PDU: *Data*.
- **Transport (TCP):** добавляет TCP-заголовок (порты, seq/ack) — PDU: *Segment*.
- **Internet (IP):** добавляет IP-заголовок (IP-адреса, TTL) — PDU: *Packet*.
- **Link (Ethernet):** добавляет Ethernet-заголовок (MAC) + trailer (FCS) — PDU: *Frame*.
- **Physical:** превращает в *Bits*.
- Матрёшка: `[ Eth hdr | IP hdr | TCP hdr | HTTP data | Eth trailer ]`.

**E2.** *(Урок 1)* **Коммутация каналов** — выделенный путь на весь сеанс (как телефон): гарантии, но расточительно (простой при молчании). **Коммутация пакетов** — данные режутся на пакеты, линии делятся всеми: эффективно и устойчиво, но без гарантий. Интернет выбрал пакеты, т.к. трафик «взрывной» (bursty) и выделять линии было бы расточительно. Плюс: эффективность/устойчивость. Минус: нет врождённых гарантий (потери, задержки, переупорядочивание).

**E3.** *(Урок 2)*
- **Потеря** → ретрансмиссии + congestion control (TCP, на концах).
- **Нарушение порядка** → sequence numbers, переупорядочивание (TCP, на получателе).
- **Дублирование** → дедупликация по номерам (TCP, на получателе).
- **Задержка** → не «лечится» полностью; смягчается буферизацией, выбором путей, размещением данных ближе (CDN).

**E4.** *(Урок 6)* Маршрутизатор снимает Ethernet-заголовок и поднимается **до L3 (IP)**, потому что решение о маршруте принимается по IP-адресу; выше (TCP/данные) ему не нужно — это для него непрозрачный груз. Он читает IP-адрес назначения, уменьшает **TTL на 1**, пересчитывает контрольную сумму, затем заново инкапсулирует пакет в **новый L2-кадр** с MAC-адресом следующего узла. Меняется: MAC-адреса (новые), TTL (−1); IP-адреса остаются прежними.

## 🎯 Бонус

Образец сильного ответа: *«L2 обеспечивает доставку кадра следующему соседу в пределах одной локальной сети по физическому MAC-адресу, который переписывается на каждом хопе; L3 обеспечивает сквозную доставку пакета через множество сетей по логическому IP-адресу, который остаётся неизменным от источника до назначения, и именно L3 принимает решения о маршрутизации».* *(Уроки 4, 6)*
