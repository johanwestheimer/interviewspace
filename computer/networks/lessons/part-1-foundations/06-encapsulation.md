## 0. Где мы и куда идём

В Уроках 3 и 5 я повторял фразу: «данные обрастают конвертами, проходя вниз по уровням». Сейчас разберём это строго: **что именно**, **на каком уровне** и **зачем** добавляется к данным, когда они спускаются по стеку у отправителя, и как всё это разбирается у получателя.

---

## 1. Главная аналогия: матрёшка из конвертов

Вспомни почтовую аналогию из Урока 2, но усложни её. Представь, что ты отправляешь секретное сообщение через цепочку посредников:

1. Ты пишешь **записку** (данные).
2. Кладёшь её в конверт и пишешь: «для отдела продаж» (**транспортный** конверт — кому из приложений).
3. Этот конверт кладёшь в конверт побольше с адресом здания компании в другом городе (**сетевой** конверт — глобальный адрес).
4. А этот — в конверт с именем курьера, который повезёт его до ближайшего почтового узла (**канальный** конверт — кто следующий сосед).

Получается **матрёшка**: записка внутри, вокруг неё — несколько вложенных конвертов, каждый со своим адресом для своего «уровня» доставки.

На той стороне получатель вскрывает конверты **в обратном порядке**: сначала внешний (курьерский), потом адрес здания, потом «отдел продаж», и только в самом конце читает записку.

🔑 **Это и есть инкапсуляция (упаковка вниз) и деинкапсуляция (распаковка вверх).** Каждый уровень добавляет свой конверт = **заголовок (header)**, не заглядывая внутрь чужого.

---

## 2. Два ключевых термина

### Инкапсуляция (encapsulation)

Процесс у **отправителя**: данные идут **сверху вниз** по стеку (Application → Link), и каждый уровень **добавляет свой заголовок** (а канальный — ещё и «хвост», trailer) к тому, что получил сверху.

🔑 Критически важная мысль: для каждого уровня всё, что пришло сверху, — это **непрозрачный груз (payload)**. Уровень IP не знает и не хочет знать, что внутри — HTTP это или почта. Он просто берёт «чёрный ящик» сверху, шлёпает свой заголовок и передаёт вниз. **Это прямое воплощение принципа разделения уровней из Урока 3.**

### Деинкапсуляция (decapsulation)

Обратный процесс у **получателя**: данные идут **снизу вверх** (Link → Application), и каждый уровень **снимает свой заголовок**, читает его, и передаёт оставшийся payload наверх.

> 💡 Симметрия: что один уровень упаковал на отправителе, **тот же уровень** распакует на получателе. Заголовок L3 пишет сетевой уровень отправителя — читает сетевой уровень получателя. Это и есть тот самый «горизонтальный разговор равных уровней» из Урока 3, теперь ты видишь его механику.

## 3. PDU: как порция данных называется на каждом уровне

Когда данные обрастают заголовками, на каждом уровне «единица данных» (PDU — Protocol Data Unit) получает своё имя. **Это любят спрашивать на собесе — выучи цепочку:**

| Уровень (TCP/IP) | Что добавляет | Имя PDU на этом уровне |
|------------------|---------------|------------------------|
| Application | данные приложения | **Data / Message** (данные) |
| Transport (TCP/UDP) | TCP/UDP-заголовок (порты, seq...) | **Segment** (TCP) / **Datagram** (UDP) |
| Internet (IP) | IP-заголовок (IP-адреса, TTL...) | **Packet** (пакет) / IP-datagram |
| Link (Ethernet) | Ethernet-заголовок + trailer (MAC, FCS) | **Frame** (кадр) |
| Physical | — превращает в сигнал | **Bits** (биты) |

🧠 **Мнемоника сверху вниз:** Data → Segment → Packet → Frame → Bits.
Многие запоминают так: «**D**o **S**ergeants **P**ay **F**or **B**eer».

> ⚠️ В быту все говорят «пакет» про что угодно. Но на собеседовании точность ценится: сегмент — это L4 (TCP), пакет — L3 (IP), кадр — L2 (Ethernet). Путать их — признак новичка.

## 4. Полная картина: как растёт «матрёшка»

Посмотрим, как HTTP-запрос обрастает заголовками, спускаясь по стеку. Каждый следующий уровень оборачивает всё предыдущее целиком:

```
Application:                          [ HTTP data ]
                                          │ вниз
Transport:                 [ TCP hdr | HTTP data ]                     ← это SEGMENT
                                          │ вниз
Internet:        [ IP hdr | TCP hdr | HTTP data ]                      ← это PACKET
                                          │ вниз
Link:    [ Eth hdr | IP hdr | TCP hdr | HTTP data | Eth trailer ]      ← это FRAME
                                          │ вниз
Physical:  10110100101110...01010  (биты в провод/радио)              ← это BITS
```

Обрати внимание на ключевые свойства:

- **Вложенность.** Заголовок TCP оказывается «внутри» по отношению к IP, а данные приложения — в самой глубине. Внешний заголовок (Ethernet) снимается первым на приёме.
- **Каждый заголовок — для своего «коллеги».** Ethernet-заголовок читает канальный уровень следующего узла; IP-заголовок — сетевой уровень конечного (или промежуточного) хоста; TCP-заголовок — транспорт конечного хоста.
- **Trailer (хвост).** Только канальный уровень (Ethernet) добавляет не только заголовок спереди, но и «хвост» сзади — поле FCS (контрольная сумма для проверки целостности кадра, см. Урок 8).

### Что в основных заголовках (превью будущих уроков)

Чтобы цепочка была не абстрактной — вот что несёт каждый конверт (детально разберём позже):

- **Ethernet-заголовок (L2):** MAC-адрес назначения и источника (Урок 8–9).
- **IP-заголовок (L3):** IP-адрес назначения и источника, TTL, protocol (Урок 14).
- **TCP-заголовок (L4):** порт источника и назначения, sequence/ack номера, флаги (Урок 23).

## 5. Что происходит на маршрутизаторе: частичная распаковка

Это тонкий и любимый на собесах момент. Когда кадр приходит на **промежуточный маршрутизатор** (не конечный хост), распаковка идёт **не до самого верха**:

```
Маршрутизатор получил кадр:
  1. Снимает Ethernet-заголовок (L2)         ← деинкапсуляция до L3
  2. Читает IP-заголовок (L3): "куда дальше?" ← принимает решение о маршруте
  3. Уменьшает TTL на 1, пересчитывает контрольную сумму
  4. Заворачивает пакет в НОВЫЙ Ethernet-кадр ← новая инкапсуляция на L2
     с MAC-адресом следующего узла
  5. Отправляет дальше
```

🔑 **Два важнейших вывода:**

1. **Маршрутизатор поднимается только до L3** (IP), читает адрес, решает куда слать — и заново упаковывает на L2. Транспортный заголовок (TCP) и данные он **не трогает** — они для него непрозрачный груз. (Поэтому маршрутизатор — устройство L3, как мы говорили в Уроке 4.)
2. **L2-заголовок (MAC) переписывается на каждом хопе**, а **L3-заголовок (IP) остаётся неизменным** на всём пути (кроме TTL).

> 🎯 Это объясняет глубокую вещь: **MAC-адреса меняются от хопа к хопу (локальная доставка соседу), а IP-адреса остаются постоянными от источника до назначения (глобальная доставка).** Если ты это понял — ты понял суть разделения L2 и L3. Запомни эту фразу, на собесе она звучит как ответ эксперта.

## 6. Overhead: цена инкапсуляции

Вспомни overhead из Урока 2 — теперь можно посчитать конкретно. Каждый заголовок — это служебные байты «налога»:

- Ethernet-заголовок + trailer: ~18 байт
- IP-заголовок: 20 байт (минимум, IPv4)
- TCP-заголовок: 20 байт (минимум)

Итого только заголовки = **~58 байт** на каждый кадр, ещё до твоих данных. Если ты передаёшь 1 байт полезных данных, эффективность чудовищна. Если передаёшь полный кадр (~1500 байт MTU) — накладные расходы ~4%, приемлемо.

🔑 Отсюда практический вывод (всплывёт в TCP и в производительности): **выгодно слать большими порциями, а не по чуть-чуть** — чтобы «размазать» фиксированный overhead заголовков на больше полезных данных. Маленькие пакеты = высокий относительный overhead.

## 7. Соберём картину воедино

- **Инкапсуляция** (у отправителя, сверху вниз): каждый уровень добавляет свой **заголовок** к payload сверху, не заглядывая внутрь. **Деинкапсуляция** (у получателя, снизу вверх): каждый уровень снимает свой заголовок.
- Что упаковал уровень N отправителя — распаковывает уровень N получателя (механика «горизонтального разговора» из Урока 3).
- **PDU по уровням:** Data → Segment → Packet → Frame → Bits.
- Получается **матрёшка** вложенных конвертов; внешний (Ethernet) снимается первым.
- На **маршрутизаторе** распаковка только до L3: **MAC-заголовок переписывается на каждом хопе**, **IP-заголовок неизменен** (меняется лишь TTL).
- Каждый заголовок — **overhead** (~58 байт TCP/IP/Eth); выгодно слать большими порциями.

## 8. Как это спрашивают на собеседовании

- **«Что такое инкапсуляция? Опиши процесс».** → Данные идут вниз по стеку, каждый уровень добавляет заголовок; на приёме — наоборот. Привести матрёшку и цепочку PDU.
- **«Как называется PDU на каждом уровне?»** → Data → Segment → Packet → Frame → Bits. Не путать сегмент/пакет/кадр.
- **«Что происходит с пакетом на маршрутизаторе?»** → Деинкапсуляция до L3, чтение IP, TTL−1, новая L2-инкапсуляция с MAC следующего узла.
- **«Меняется ли IP-адрес/MAC-адрес по пути?»** → IP остаётся постоянным от источника до получателя; MAC переписывается на каждом хопе. (Сильный ответ!)
- **«Почему невыгодно слать много мелких пакетов?»** → Фиксированный overhead заголовков на каждый пакет; относительные накладные расходы растут.
- **«Видит ли маршрутизатор TCP-заголовок / данные?»** → Обычно нет — он работает до L3; для него транспорт и данные — непрозрачный груз (если это не L4+ устройство вроде файрвола/балансировщика).

> 🎯 **Уровень топ-компаний:** связать инкапсуляцию с принципом уровней (каждый видит только свой конверт) и вывести из неё фундаментальное «MAC меняется, IP постоянен». Это показывает, что ты понимаешь *почему*, а не заучил список.
